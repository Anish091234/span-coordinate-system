SPAN — Solar Positional Address Notation

A hierarchical spacecraft addressing system built on a continuous J2000 ICRF backbone. One address, any body, any scale.

Background

A spacecraft moving through the solar system crosses multiple coordinate regimes: ECI at launch, heliocentric ecliptic during cruise, body-centred for orbit insertion, geodetic for landing. Each frame lives in a separate document and gets re-expressed manually at each handoff. There's no standard notation that tracks which body currently owns a vehicle or at what scale, so trajectory files go ambiguous and ground-system interfaces need constant reconciliation.

Approach

SPAN gives every craft one live address. The dotted path encodes nesting (like an IP address for the solar system) and the leaf coordinates use whichever scheme fits the current scale. Cross a sphere-of-influence boundary and the address updates. The math underneath stays in ICRF the whole time.

SOL.TER.LUN  ⟨ X +812.4  Y −1203.0  Z +44.6 km ⟩
SOL = heliocentric root  ·  TER = inside Earth's SOI  ·  LUN = active frame (Luna's SOI)  ·  ⟨ X Y Z ⟩ = Cartesian in Luna frame
SOL
Heliocentric · J2000 ecliptic
λ · β · r (AU)
Ecliptic longitude, latitude, and range in AU. Fits interplanetary cruise and matches JPL Horizons vector output directly.
PLANET
Body-centred · spherical
r · Az · El (km)
Range, azimuth, and elevation from the body centre. Used for orbit insertion burns and rendezvous manoeuvres.
MOON
Local · Cartesian
X · Y · Z (km)
Right-handed Cartesian in km. Covers terminal descent, proximity ops, and docking at the metre level.

ICRF Backbone & Frame Continuity

SPAN is a labeling layer over a continuous J2000 ICRF coordinate backbone. When a vehicle crosses a sphere-of-influence boundary and the address changes from SOL.TER to SOL.TER.LUN, the underlying ICRF Cartesian position vector is unchanged. It's a relabeling, not a coordinate jump. Each SPAN address maps one-to-one to an ICRF triple at a given epoch, so it's compatible with SPICE kernels, STK, and anything else that speaks J2000.

Ephemeris · NASA JPL Horizons

Planet and Moon positions are fetched live from NASA JPL's Horizons API (ssd.jpl.nasa.gov/api/horizons.api) on page load. The API returns J2000 ICRF Keplerian elements (a, e, i, Ω, ω, ν) for each body, which get converted to heliocentric ecliptic Cartesian via the rotation Rz(Ω)·Rx(i)·Rz(ω) to extract the real orbital longitude. Planets are shown at their actual positions today, verifiable against Horizons directly. Hit Elements in the readout to see the full Keplerian set for any body in the address chain.

How to Use
Address bar
Live SPAN address updates as you move. Click any crumb to jump to that body's orbit.
Elements toggle
Switch the readout from position coordinates to Keplerian elements (a, e, i, Ω, ω, ν) sourced from Horizons.
Mission presets
Pick a mission and step through each phase. Each step places the craft and explains what the address is doing.
Pilot console
Drag sliders to fly manually. SOI handoffs happen automatically as you cross influence boundaries.
Orbit playback
Press Orbits to animate all bodies at 1×–8× time compression. Positions start from today's real Horizons values.
Surface regime
Get within one body-radius of a surface to switch into geodetic mode: lat/long/altitude with a live ground-track dot.

SOI radius: a·(m/M)2/5. Solar-system view uses √ radial compression. Craft orbital elements marked "circular approx" assume e≈0 since velocity is not tracked; a full state vector propagator would be needed for anything precise. Horizons data is fetched once at page load and positions drift from there during playback.
