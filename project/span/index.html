SPAN — Solar Positional Address Notation

A hierarchical spacecraft addressing system that unifies multi-body navigation under a single, human-readable notation — and a continuous J2000 ICRF mathematical backbone.

The Problem

Every operational spacecraft toggles between incompatible coordinate frames as it moves through the solar system: ECI for launch, heliocentric ecliptic for cruise, body-centred for orbit insertion, geodetic for landing. Each frame lives in a separate reference document and is re-expressed manually at each handoff. There is no standard notation that captures which body currently owns a vehicle and at what scale — making multi-mission trajectory files ambiguous and ground-system interfaces fragile.

The Solution

SPAN gives every craft one live address. The dotted path encodes nesting — like an IP address for the solar system — and the leaf coordinates use whichever scheme is physically natural at the current scale. The address is always valid and always unambiguous.

SOL.TER.LUN  ⟨ X +812.4  Y −1203.0  Z +44.6 km ⟩
SOL = heliocentric root  ·  TER = inside Earth's SOI  ·  LUN = active frame (Luna's SOI)  ·  ⟨ X Y Z ⟩ = Cartesian in Luna frame
SOL
Heliocentric · J2000 ecliptic
λ · β · r (AU)
Ecliptic longitude, latitude, and range in AU. Natural frame for interplanetary cruise — matches JPL Horizons vector output directly.
PLANET
Body-centred · spherical
r · Az · El (km)
Range, azimuth, and elevation from the body centre. The standard language of orbit insertion burns and rendezvous manoeuvres.
MOON
Local · Cartesian
X · Y · Z (km)
Right-handed Cartesian in km. Precision proximity ops, terminal descent, and docking — where every metre counts.

ICRF Backbone & Frame Continuity

SPAN is a labeling abstraction layered over a continuous J2000 ICRF coordinate backbone — not a patchwork of disconnected frames. When a vehicle crosses a sphere-of-influence boundary and the address updates from SOL.TER to SOL.TER.LUN, the underlying ICRF Cartesian position vector does not change. The transition is a relabeling event — not a physical discontinuity. Every SPAN address maps bijectively to a unique ICRF triple at a given epoch. This means SPAN addresses are directly interoperable with SPICE kernels, STK, and any software that understands J2000.

Real Ephemeris · NASA JPL Horizons

Planet and Moon positions are fetched live from NASA JPL's Horizons System API (ssd.jpl.nasa.gov/api/horizons.api) on every page load. The API returns J2000 ICRF Keplerian elements (a, e, i, Ω, ω, ν) for each body. These are converted to heliocentric ecliptic Cartesian coordinates via the standard rotation Rz(Ω)·Rx(i)·Rz(ω) and used to set each planet's true orbital longitude. The result: every planet shown is where it actually is today, verifiable against Horizons directly. Toggle Elements in the readout panel to inspect the full Keplerian set for each body in the address chain.

How to Use
Address bar
Live SPAN address updates as you move. Click any crumb to jump to that body's orbit.
Elements toggle
Switch the readout from position coordinates to Keplerian elements (a, e, i, Ω, ω, ν) sourced from Horizons.
Mission presets
Step through Dragon ISS rendezvous or Starship lunar landing phase by phase, with engineering commentary.
Pilot console
Drag sliders to fly manually. SOI handoffs trigger automatically when you cross a sphere-of-influence boundary.
Orbit playback
Press Orbits to animate all bodies at 1×–8× time compression. Positions start from today's real Horizons values.
Surface regime
Descend within one body-radius of any planet or moon to engage geodetic mode — lat/long/altitude with a live ground-track.

SOI radius: a·(m/M)2/5. Solar-system view uses √ radial compression so all planets are visible simultaneously. Craft orbital elements labelled "circular approx" assume e≈0 — velocity is not tracked; full elements require a state vector propagator. Horizons data is fetched at page load; positions drift at orbit playback speed thereafter.
